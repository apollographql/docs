This reference lists the rules that you can enforce with [GraphOS schema linting](./schema-linter), along with the code that GraphOS returns for each rule violation.

## Naming rules

These rules enforce naming conventions. Rules are categorized by the part(s) of your schema that they correspond to.

### Fields

<RuleExpansionPanel
  title="FIELD_NAMES_SHOULD_BE_CAMEL_CASE"
  whatItDoes="Checks that all field names use `camelCase`."
  rationale="`camelCase` is a convention for field names in GraphQL."
>

The following example violates the rule:

```graphql title="‚ùå schema.graphql" disableCopy=true showLineNumbers=false
type User {
  # highlight-start
  FirstName: String! # PascalCase
  # highlight-end
}
```

<br/>
Use instead:
<br/>

```graphql title="‚úÖ schema.graphql" disableCopy=true showLineNumbers=false
type User {
  # highlight-start
  firstName: String # camelCase
  # highlight-end
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="RESTY_FIELD_NAMES"
  whatItDoes="Checks that a field's name doesn't start with any of the following verbs: `get`, `list`, `post`, `put`, `patch`."
  rationale="Fields should not start with a verb, except for mutations. For mutation fields, it's best to use a verb that describes the specific action being performed, for example, `create`, `delete`, or `edit`.">

The following example violates the rule:

```graphql title="‚ùå schema.graphql" disableCopy=true showLineNumbers=false
type Query {
  getUsers: [User!]! # highlight-line
}
```

<br/>
Use instead:
<br/>

```graphql title="‚úÖ schema.graphql" disableCopy=true showLineNumbers=false
type Query {
  users: [User!]! # highlight-line
}
```

</RuleExpansionPanel>

### Types

These rules apply to all types that appear in a GraphQL schema, including:

- Objects
- Interfaces
- Inputs
- Enums
- Unions

<RuleExpansionPanel
  title="ENUM_USED_AS_INPUT_WITHOUT_SUFFIX"
  whatItDoes="If an enum type is used as an input argument, checks that its name uses the suffix `Input`."
  rationale='Ending input arguments with `Input` distinguishes inputs from "output" types like objects.'
>

The following example violates the rule:

```graphql title="‚ùå schema.graphql" disableCopy=true showLineNumbers=false

  VIEWER
}

```

<br/>

```graphql title="‚úÖ schema.graphql" disableCopy=true showLineNumbers=false
enum RoleInput { # highlight-line
  EDITOR
  VIEWER
}

type Query {
  users(role: RoleInput): [User!]! # highlight-line
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="ENUM_USED_AS_OUTPUT_DESPITE_SUFFIX"
  whatItDoes="If an enum is used as the return type of a non-input field, checks that its name doesn't use the suffix `Input`."

```graphql title="‚ùå schema.graphql" disableCopy=true showLineNumbers=false
  VIEWER

}
```

<br/>
Use instead:
<br/>

```graphql title="‚úÖ schema.graphql" disableCopy=true showLineNumbers=false

  EDITOR
  VIEWER
}

type Query {
  userRole(userId: ID!): Role # highlight-line
}
```

</RuleExpansionPanel>

## Other rules

These rules define conventions for the entire schema and directive usage outside of composition.

### Schema

These rules apply to the entire schema.

<RuleExpansionPanel
  title="DOES_NOT_PARSE"
  whatItDoes="Checks for malformed GraphQL schemas. To resolve, check your schema for syntax errors."
/>

<RuleExpansionPanel
  title="ALL_ELEMENTS_REQUIRE_DESCRIPTION"
  whatItDoes="Checks that each element in the schema includes a [description](/apollo-server/schema/schema/#descriptions-docstrings)."
  rationale="Descriptions document your GraphQL schema so consumers of your graph can easily discover fields and learn how to use them. See [examples](/apollo-server/schema/schema/#descriptions-docstrings)."
>

The following example violates the rule:

```graphql title="‚ùå schema.graphql" disableCopy=true showLineNumbers=false
type User {
  username: String!
}
```

<br/>
Use instead:
<br/>

```graphql title="‚úÖ schema.graphql" disableCopy=true showLineNumbers=false
"Represents a user"
type User {
 "A username must be [8-64] characters."
 username: String!
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="DEFINED_TYPES_ARE_USED"
  whatItDoes="Checks that every type defined in a schema is be used at least once."
  rationale="Unused types waste resources and should be immediately removed once they've been refactored out of a schema."
>

The following example violates the rule:

```graphql title="‚ùå schema.graphql" disableCopy=true showLineNumbers=false
type SomeUnusedType { # Also fails the TYPE_SUFFIX rule!
  name: String!
}

type AnActuallyUsedType {
  name: String!
}

type Query {
  hello: String!
  title: AnActuallyUsedType
}
```

<br/>
Use instead:
<br/>

```graphql title="‚úÖ schema.graphql" disableCopy=true showLineNumbers=false
type Book {
  title: String!
}

type Query {
  books: [Book!]!
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="QUERY_DOCUMENT_DECLARATION"
  whatItDoes="Checks that schemas don't define operations, such as queries and mutations."
  rationale="Operations should be defined on the client side, not within schemas. Schemas define types, fields, and their relationships. Operations specify what data to fetch or change. Since what to fetch or change is a client concern, this separation allows for a clean and modular architecture, making it easier to manage and evolve the API over time."
>

The following example violates the rule:

```graphql title="‚ùå schema.graphql" disableCopy=true showLineNumbers=false
type Query {
  users: [User!]!
}

#highlight-start
query GetUsers {
  # Don't define operations in a schema document
  users {
    id
  }
}
#highlight-end
```

</RuleExpansionPanel>

### Directives

<RuleExpansionPanel
  title="CONTACT_DIRECTIVE_MISSING"
  whatItDoes="Checks that subgraph schemas always provide owner contact details via the `@contact` directive."
  rationale="You can use the `@contact` directive to add your team's contact info to a subgraph schema. This info is displayed in GraphOS Studio, which helps other teams know who to contact for assistance with the subgraph. [Learn more](../graphs/federated-graphs#contact-info-for-subgraphs)."
>

This example shows correct `@contact` directive usage:
<br/>

```graphql title="‚úÖ schema.graphql" disableCopy=true showLineNumbers=false
directive @contact(
  "Contact title of the subgraph owner"
  name: String!
  "URL where the subgraph's owner can be reached"
  url: String
  "Other relevant notes can be included here; supports markdown links"
  description: String
) on SCHEMA

extend schema
  @contact(
    name: "Products Team"
    url: "https://myteam.slack.com/archives/teams-chat-room-url"
    description: "Send urgent issues to [#oncall](https://yourteam.slack.com/archives/oncall)."
  )
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="DEPRECATED_DIRECTIVE_MISSING_REASON"
  whatItDoes="Checks that the `@deprecated` directive always includes a `reason` argument."
  rationale="The `reason` argument can help graph consumers understand which field to use instead of the `@deprecated` one. "
>

The following example violates the rule:

```graphql title="‚ùå schema.graphql" disableCopy=true showLineNumbers=false
type Product {
  title: String @deprecated #highlight-line
  name: String!
}
```

<br/>
Use instead:
<br/>

```graphql title="‚úÖ schema.graphql" disableCopy=true showLineNumbers=false
type Product {
  title: String @deprecated(reason: "Use Product.name instead") #highlight-line
  name: String!
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="TAG_DIRECTIVE_USES_UNKNOWN_NAME"
  whatItDoes="Checks that the [`@tag`](/federation/federated-types/federated-directives/#tag) directive uses an approved value for its `name` argument. You specify approved values in [GraphOS Studio](./schema-linter/#linter-configuration)."
  rationale="This directive is used most commonly with [GraphOS contracts](./contracts). Validating `@tag` names ensures contracts work as intended."
>

</RuleExpansionPanel>

## Custom rules

The schema linter is [configurable](./schema-linter#linter-configuration) with the predefined rules documented above.
Custom rule creation isn't currently supported but is under consideration.
