This reference lists the rules that you can enforce with [GraphOS schema linting](./schema-linter/), along with the code that GraphOS returns for each rule violation.

## Naming rules

These rules enforce naming conventions. Rules are categorized by the part(s) of your schema that they correspond to.

### Fields

<RuleExpansionPanel
  title="FIELD_NAMES_SHOULD_BE_CAMEL_CASE"
  whatItDoes="Checks that all field names use `camelCase`."
  rationale="`camelCase` is a common convention for field names in GraphQL."
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type User {
  # highlight-start
  FirstName: String! # PascalCase
  # highlight-end
}
```

<br/>
Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type User {
  # highlight-start
  firstName: String # camelCase
  # highlight-end
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="RESTY_FIELD_NAMES"
  whatItDoes="Checks that field's name don't start with any of the following verbs: `get`, `list`, `post`, `put`, `patch`."
  rationale="Most fields should not start with any verb, with the exception of mutation fields. For mutation fields, it's best to use a verb that more specifically describes the action being performed, for example, `create`, `delete`, or `edit`.">

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type Query {
  getUsers: [User!]! # highlight-line
}
```

<br/>
Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type Query {
  users: [User!]! # highlight-line
}
```

</RuleExpansionPanel>

### Types

These rules apply to _all_ types that appear in a GraphQL schema, including:

- Objects
- Interfaces
- Inputs
- Enums
- Unions

<RuleExpansionPanel
  title="TYPE_NAMES_SHOULD_BE_PASCAL_CASE"
  whatItDoes="Checks that all type names use `PascalCase`."
  rationale="`PascalCase` is a common convention for type names in GraphQL."
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
# highlight-start
type streamingService { # camelCase
  # highlight-end
  id: ID!
}
```

<br/>
Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
# highlight-start
type StreamingService { # PascalCase
  # highlight-end
  id: ID!
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="TYPE_PREFIX"
  whatItDoes="Checks that type names don't use the prefix `Type`."
  rationale="Starting type names with `Type` is redundant. It makes your schema definitions longer than necessary and your schema less readable."
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false

  title: String!
}
```

<br/>
Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type Book { # highlight-line
  title: String!
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="TYPE_SUFFIX"
  whatItDoes="Checks that type names don't use the suffix `Type`."
  rationale="Ending type names with `Type` is redundant. It makes your schema definitions longer than necessary and your schema less readable."

The following example violates the rule:

### Inputs and arguments

<RuleExpansionPanel
  title="INPUT_ARGUMENT_NAMES_SHOULD_BE_CAMEL_CASE"
  whatItDoes="Checks that argument names use `camelCase`."
  rationale="`camelCase` is a common convention for argument names in GraphQL."
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
  #highlight-end
```

<br/>

<br/>


</RuleExpansionPanel>

<RuleExpansionPanel
  title="INPUT_TYPE_SUFFIX"
  whatItDoes="Checks that input type names use the suffix `Input`."
  rationale='Ending input type names with `Input` distinguishes input types from "output" types like objects.'
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false

  title: String!
  content: String!
}
```

<br/>
Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false

  title: String!
  content: String!
```
```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="ENUM_USED_AS_INPUT_WITHOUT_SUFFIX"
  whatItDoes="If an enum type is used as an input argument, checks that its name uses the suffix `Input`."
  rationale='Ending input arguments with `Input` distinguishes inputs from "output" types like objects.'
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
enum Role { # highlight-line
  EDITOR
  VIEWER
}

type Query {
  users(role: Role): [User!]! # highlight-line
}
```

<br/>
Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
enum RoleInput { # highlight-line
  EDITOR
  VIEWER
}

type Query {
  users(role: RoleInput): [User!]! # highlight-line
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="ENUM_USED_AS_OUTPUT_DESPITE_SUFFIX"
  whatItDoes="If an enum is used as the return type of a _non_-input field, checks that its name doesn't use the suffix `Input`."
  rationale="Including the suffix `Input` on an output type is confusing."
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
enum RoleInput { # highlight-line
  EDITOR
  VIEWER
}

type Query {
  userRole(userId: ID!): RoleInput # highlight-line
}
```

<br/>
Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
enum Role { # highlight-line
  EDITOR
  VIEWER
}

type Query {
  userRole(userId: ID!): Role # highlight-line
}
```

</RuleExpansionPanel>

## Other rules

These rules define conventions for the entire schema and directive usage outside of composition.

### Schema

These rules apply to the entire schema.

<RuleExpansionPanel
  title="DOES_NOT_PARSE"
  whatItDoes="Checks for malformed GraphQL schemas. Check your schema for any syntax errors."
/>

<RuleExpansionPanel
  title="ALL_ELEMENTS_REQUIRE_DESCRIPTION"
  whatItDoes="Checks that each element in the schema includes a [description](/apollo-server/schema/schema/#descriptions-docstrings)."
  rationale="Descriptions serve as documentation for your GraphQL schema. They make it easier for consumers of your graph discover fields and learn how to use them. Check out [examples](/apollo-server/schema/schema/#descriptions-docstrings)."
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type User {
  username: String!
}
```

<br/>
Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
"Represents a user"
type User {
  "The user's username"
  username: String!
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="DEFINED_TYPES_ARE_USED"
  whatItDoes="Checks that every type defined in a schema is be used at least once."
  rationale="Unused types commonly appear after you refactor other parts of your schema. You should remove unused types to prevent confusion."
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type SomeUnusedType { # Also fails the TYPE_SUFFIX rule!
  name: String!
}

type AnActuallyUsedType {
  name: String!
}

type Query {
  hello: String!
  title: AnActuallyUsedType
}
```

<br/>
Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type Book {
  title: String!
}

type Query {
  books: [Book!]!
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="QUERY_DOCUMENT_DECLARATION"
  whatItDoes="Checks that schemas don't define operations such as queries and mutations."
  rationale="Schemas define types, fields, and their relationships. Operations specify what data to fetch or change. Since what to fetch or change is a client concern, operations should be defined on the client side. This separation allows for a clean and modular architecture, making it easier to manage and evolve the API over time."
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type Query {
  users: [User!]!
}

#highlight-start
query GetUsers {
  # Don't define operations in a schema document
  users {
    id
  }
}
#highlight-end
```

</RuleExpansionPanel>

### Directives

<RuleExpansionPanel
  title="CONTACT_DIRECTIVE_MISSING"
  whatItDoes="Checks that subgraph schemas always provide owner contact details via the `@contact` directive."
  rationale="You can use the `@contact` directive to add your team's contact info to a subgraph schema. This info is displayed in GraphOS Studio, which helps other teams know who to contact for assistance with the subgraph. [Learn more](../graphs/federated-graphs#contact-info-for-subgraphs)."
>

Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
directive @contact(
  "Contact title of the subgraph owner"
  name: String!
  "URL where the subgraph's owner can be reached"
  url: String
  "Other relevant notes can be included here; supports markdown links"
  description: String
) on SCHEMA

extend schema
  @contact(
    name: "Products Team"
    url: "https://myteam.slack.com/archives/teams-chat-room-url"
    description: "Send urgent issues to [#oncall](https://yourteam.slack.com/archives/oncall)."
  )
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="DEPRECATED_DIRECTIVE_MISSING_REASON"
  whatItDoes="Checks that the `@deprecated` directive always includes a `reason` argument."
  rationale="The `reason` argument can help graph consumers understand which field to use instead. "
>

The following example violates the rule:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type Product {
  title: String @deprecated #highlight-line
  name: String!
}
```

<br/>
Use instead:
<br/>

```graphql title="schema.graphql" disableCopy=true showLineNumbers=false
type Product {
  title: String @deprecated(reason: "Use Product.name instead") #highlight-line
  name: String!
}
```

</RuleExpansionPanel>

<RuleExpansionPanel
  title="TAG_DIRECTIVE_USES_UNKNOWN_NAME"
  whatItDoes="Checks that the [`@tag`](/federation/federated-types/federated-directives/#tag) directive uses an approved value for its `name` argument. You specify approved values in [GraphOS Studio](./schema-linter/#configuring-the-linter)."
  rationale="This directive is used most commonly with [GraphOS contracts](../contracts). Validating `@tag` names ensures contracts work as intended."
>

</RuleExpansionPanel>
